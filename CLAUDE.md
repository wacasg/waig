# WAIG — 「店長を救え！」開発ルール（CLAUDE.md）

AI・デジタルビジネス学科のオープンキャンパス／出張授業で使う**体験用ミニゲーム**。
来場した高校生が「AIと一緒に意思決定する」体験を 3 分で味わうのが目的。

- 公開URL: https://wacasg.github.io/waig/
- リポジトリ: https://github.com/wacasg/waig （`main` ブランチの root から GitHub Pages 配信）
- 配信物: `index.html`（**本番**） / `tencho_save_me.html`（旧オリジナルのバックアップ・触らない）

---

## 0. 最重要の心構え

このゲームの**主役は来場する高校生**。「むずかしい用語を避け、3 分で達成感」が絶対基準。
ロジックの正しさより**体験の手触り**を優先する。迷ったら「高校生がワクワクするか？」で判断する。

---

## 1. アーキテクチャと制約

- 現状は **`index.html` 単一ファイル**（HTML+CSS+JS、外部依存ゼロ）。CDN・npm・ビルド不要。
  - この「ファイル単体でブラウザにドロップすれば動く」性質は**当面維持**する。外部 `<script src>` / `<link href>`（CDN・Webフォント・解析タグ等）を**追加しない**。
  - 例外: 将来規模が大きくなれば JS/CSS 分割やビルド導入も検討可（その場合は必ず事前に方針合意 → README とこの章を更新）。今は単一ファイル前提で進める。
- 編集対象は原則 `index.html` のみ。`tencho_save_me.html` は履歴保全用なので変更しない。

## 2. ゲーム機構（コードを変える前に把握する）

`index.html` 内 `<script>` に全ロジックがある。主要素：

| 要素 | 場所 | 内容 |
|---|---|---|
| `CONFIG` | 冒頭 | 先生が書き換える設定（後述）。**ここ以外を触らず調整できる範囲を広く保つ** |
| メーター | `METER_INFO` | `kya`=集客 / `tsu`=使いやすさ / `kai`=買いたい度（各 0–100） |
| 売上計算 | `calcYen(s)` | `(0.34*kya + 0.33*tsu + 0.33*kai) × (0.6 + 0.4×min/100) × 7700` を四捨五入。**弱点（最小メーター）が足を引っぱる**設計 |
| 開始市場 | `startMarket()` | 1つ強め(62–74)＋2つ弱め(44–55)をシャッフル。毎プレイ変化 |
| イベント | `TURN_POOLS` | 3スロット（①導入／②中盤／③ラスト）×各3候補。ゲームごとに1つずつランダム選択（`rollTurns`）|
| 選択肢 | 各 event の `actions` | `A/B/C` の3択。`eff`（メーター増分）と `aiRec`（AI推奨の選択肢）を持つ |
| 結果診断 | `diagnose()` | `target` 到達で `cleared`、最小メーター≥65 で `balanced`。4タイプ判定＋ランク S/A/B/C |

### CONFIG（先生が書き換える場所）
```
schoolName / date / place / applyText … 出前授業バナー文言
applyURL    … 申込フォームURL（空ならボタン非表示）
realGameURL … 「本物のゲームを見る」リンク（空なら非表示）
target      … 売上目標（円）。現在 520000。難易度の主ツマミ
```

### 設計の不変条件（壊さない）
- 1ゲーム＝**3ターン**。各スロットの3候補は**役割と効果量をそろえる**（どれを引いても難易度が大きく変わらない）。
- `aiRec` は「セオリー通りの無難な手」。**必ずしも最適ではない**のが教育的な肝（鵜呑みの罠を体験させる）。
- 各 event の `actions` は **A/B/C の3つ**、それぞれ別メーターを主に伸ばす（集客寄り／使いやすさ寄り／買いたい度寄り）。
- メーターは `clamp` で 0–100 に収まる。`eff` 追加時もこの範囲前提。

## 3. バランス調整の流儀

数値（`target`・各 `eff`・開始市場レンジ）を変えたら、**体感ではなくシミュレーションで確認**する。

```bash
python3 scripts/balance_sim.py            # 現状のクリア率・平均売上・戦略別勝率を出力
python3 scripts/balance_sim.py --games 50000
```

目安（崩したくないライン）:
- 「AI全追従」戦略のクリア率は **おおむね 45–70%**（簡単すぎず、鵜呑みだと時々落ちる）。
- 「弱点を埋める最適戦略」は **高クリア率（85%+）**。考えた子が報われる。
- ランク分布が S に偏りすぎ／C に偏りすぎないこと。

詳細手順は `/balance-report` コマンド、または `game-balance` スキルを使う。

## 4. コピー・教育監修の基準

- 対象は高校生。**専門用語は使わない／使うなら必ず言い換える**（例: CVR→「買いたい度」、UX→「使いやすさ」）。
- AI のセリフ（`ai` フィールド）は「断定しすぎない・根拠を一言添える」。AIを神格化も否定もしない。
- オープンキャンパス／出前授業の案内（バナー・申込文言）は**勧誘色を強くしすぎない**。体験の余韻を壊さない。
- 学科名・リンク先（日本経済大学 AI・DB学科ページ等）は正式名称・正しいURLを使い、外部リンクは別ウィンドウ（`target="_blank" rel="noopener"`）。

## 5. 開発フロー

```bash
# ローカル確認（ビルド不要）
python3 -m http.server 8765   # → http://localhost:8765/
#   または /preview コマンド

# デプロイ（main push で数分後に Pages 反映）
git add index.html
git commit -m "変更内容"
git push origin main
#   または /deploy コマンド
```

- 直接 `main` で作業して良い小さな個人プロジェクト。ただし**push 前に必ずローカルで一通りプレイ**（3ターン→結果まで）し、コンソールエラーが無いこと、`A/B/C` 全パスが動くことを確認。
- 大きめの変更は `git switch -c feat/...` でブランチを切ってから。

## 6. このリポジトリの開発支援（.claude/）

`.claude/` は **Git 管理下**（チーム共有）。中身:

- **agents/** … `game-designer`（機能・体験）/ `copy-edu-reviewer`（文言・教育監修）/ `balance-analyst`（数値）
- **commands/** … `/preview` `/deploy` `/balance-report` `/new-event`
- **skills/** … `game-balance`（数値チューニング支援＋シミュレータ）
- **hooks/** … session-start（pull＋状況）/ post-edit-guard（単一HTML・外部依存の混入を検知）
- **MCP** … 自動プレイテスト/スクショ用に Playwright（`.mcp.json`）。プレビューは Claude_Preview、運用解析は google-analytics（いずれも接続済みを利用）

`.claude/settings.local.json` は各自ローカル用（Git 管理外）。
